# Исследование стека для Telegram RAG-бота малого бизнеса

Актуально на **1 марта 2026**.

## Контекст задачи

Нужно реализовать Telegram-бота для малого бизнеса, который:

- отвечает на вопросы клиентов по базе знаний компании;
- собирает заявки в формате `имя + телефон + вопрос`;
- уведомляет владельца о новых заявках;
- работает 24/7 без сервера на старте, с локальным запуском.

Допущение: RAG выбран осознанно, даже если для части FAQ-задач это архитектурно избыточно.

---

## Общие выводы по рынку 2025-2026

К 2025-2026 для подобных решений закрепились следующие практики:

- для MVP чаще используют **long polling**, а не webhook;
- для поиска по знаниям стандартом становится **hybrid retrieval**: векторный поиск + keyword/BM25/FTS;
- сбор заявок лучше делать **детерминированно**, отдельно от LLM-логики;
- ответ модели желательно ограничивать найденным контекстом и вводить **fallback**, если уверенность низкая;
- полезно сразу закладывать **метаданные документов**: тип, категория, услуга, филиал, версия, дата обновления.

Важно: “работа без сервера” на старте означает не отсутствие постоянно работающей машины, а отсутствие отдельной облачной инфраструктуры. Для long polling нужен постоянно включенный ноутбук, мини-ПК или домашний сервер.

## 1. Язык и фреймворк для Telegram-бота

### Вариант 1. Python + aiogram 3

**Плюсы**

- современный асинхронный стек;
- удобные routers, middleware, filters, FSM;
- хорошо подходит для production-пути и расширения логики;
- сильная экосистема Python для RAG, ETL и ML.

**Минусы**

- нужен базовый комфорт с `async` / `await`;
- для новичков структура может быть чуть сложнее, чем у более “tutorial-first” библиотек.

**Сложность освоения:** средняя.

### Вариант 2. Python + python-telegram-bot

**Плюсы**

- очень понятный вход;
- сильная документация и много примеров;
- удобно делать формы, диалоги и пошаговые сценарии.

**Минусы**

- в 2026 выглядит чуть менее “инженерно удобным” для сложных async-сценариев, чем aiogram;
- обновления Bot API обычно ощущаются чуть менее быстро, чем у самых агрессивно развиваемых альтернатив.

**Сложность освоения:** низкая-средняя.

### Вариант 3. TypeScript + grammY

**Плюсы**

- отличный DX для команд на TypeScript;
- хорошие типы и plugin ecosystem;
- удобно масштабировать бот-логику в более сложную платформу.

**Минусы**

- если команда не сидит в Node/TS, вход дороже;
- для RAG-части часть команд всё равно часто тяготеет к Python-экосистеме.

**Сложность освоения:** средняя.

### Вывод по компоненту

Если команда ближе к Python, базовый выбор: **aiogram 3**.  
Если приоритетом является самый мягкий onboarding, можно выбрать **python-telegram-bot**.  
Если команда уже TypeScript-first, разумен **grammY**.

## 2. Хранение и поиск по базе знаний

### Вариант 1. PostgreSQL + pgvector + Full Text Search

**Плюсы**

- одна БД и для заявок, и для базы знаний, и для метаданных;
- зрелая SQL-экосистема;
- можно сочетать semantic search и keyword search;
- удобный путь к фильтрации и отчетности;
- проще объяснить и поддерживать команде с разным опытом.

**Минусы**

- требует аккуратно продумать hybrid ranking;
- как pure vector engine уступает специализированным системам на больших масштабах.

**Сложность освоения:** средняя.

### Вариант 2. Qdrant

**Плюсы**

- один из самых сильных вариантов под RAG;
- хорош для dense, sparse и hybrid retrieval;
- есть local mode и понятный Python client;
- легко масштабируется дальше.

**Минусы**

- часто добавляет второе хранилище рядом с основной реляционной БД;
- операционно сложнее, чем “всё в Postgres”.

**Сложность освоения:** средняя.

### Вариант 3. Weaviate

**Плюсы**

- богато оснащен для retrieval-задач;
- есть hybrid search и готовые возможности вокруг RAG.

**Минусы**

- заметно тяжелее для малого MVP;
- выше инфраструктурная и концептуальная сложность.

**Сложность освоения:** высокая.

### Вариант 4. LanceDB

**Плюсы**

- local-first;
- очень удобен для быстрого прототипа без сложной infra;
- нормален для локального старта и экспериментов.

**Минусы**

- для долгосрочного production-пути команды чаще переходят к Postgres/Qdrant;
- меньше “организационной привычности” для обычной backend-команды.

**Сложность освоения:** низкая-средняя.

### Альтернативы без классической vector DB

#### PostgreSQL FTS без векторов

**Плюсы**

- очень простая реализация;
- дешево и прозрачно;
- хорошо работает на FAQ с точными формулировками.

**Минусы**

- это уже не полноценный semantic RAG;
- хуже отрабатывает перефразированные и “человеческие” запросы.

**Сложность освоения:** низкая.

#### SQLite + FTS5

**Плюсы**

- один файл;
- почти нулевая инфраструктура;
- идеально для локального прототипа.

**Минусы**

- слабый путь роста;
- ограничения по одновременной работе и развитию системы.

**Сложность освоения:** низкая.

#### Managed retrieval через LLM-провайдера

**Плюсы**

- минимум кода и infra;
- быстрый старт.

**Минусы**

- vendor lock-in;
- меньше контроля над retrieval;
- хуже для локального и воспроизводимого развития архитектуры.

**Сложность освоения:** низкая.

### Вывод по компоненту

Для малого бизнеса с реальной перспективой роста оптимален **PostgreSQL + pgvector + FTS**.  
Для сверхбыстрого локального прототипа подходят **Qdrant Local** или **LanceDB**.

## 3. LLM для генерации ответов

### Вариант 1. OpenAI GPT-5 mini / GPT-4.1

**Плюсы**

- сильное качество ответов;
- хорошие structured outputs и tool-calling возможности;
- удобно сочетать с embeddings и retrieval-логикой;
- быстрый старт и много примеров.

**Минусы**

- облачный провайдер;
- зависимость от внешнего API.

**Сложность освоения:** низкая.

### Вариант 2. Anthropic Claude Sonnet 4

**Плюсы**

- очень сильное качество диалога;
- хорошо держит длинный контекст;
- хорошо подходит для вежливых customer-facing ответов.

**Минусы**

- экосистема практических шаблонов для малого Telegram RAG-бэкенда чуть менее “дефолтная”, чем у OpenAI в Python-среде.

**Сложность освоения:** низкая-средняя.

### Вариант 3. Google Gemini 2.5 Flash

**Плюсы**

- хорошее соотношение цены и производительности;
- большой контекст;
- развитые function/tool сценарии.

**Минусы**

- для типового малого бота стек и примеры часто менее привычны, чем OpenAI + Python.

**Сложность освоения:** низкая-средняя.

### Вариант 4. Mistral Small 3.2 / Medium 3.1

**Плюсы**

- хороший price/performance;
- разумный вариант для европейского контекста и альтернативы крупным hyperscaler-экосистемам.

**Минусы**

- меньше стандартных рецептов и туториалов именно под такой кейс.

**Сложность освоения:** средняя.

### Вариант 5. Локальные модели через Ollama

Примеры: `gpt-oss 20b`, `Qwen3 8b`, `Qwen3 30b`.

**Плюсы**

- данные остаются локально;
- можно обойтись без облачного LLM на части сценариев;
- удобно для privacy-first экспериментов.

**Минусы**

- требования к железу;
- заметно больше работы по latency, качеству и стабильности;
- для старта это почти всегда усложнение.

**Сложность освоения:** средняя-высокая.

### Вывод по компоненту

Для первого production-MVP лучший выбор: **OpenAI GPT-5 mini**.  
Если нужен quality-first вариант и команда готова к альтернативному провайдеру, можно смотреть на **Claude Sonnet 4**.  
Локальные модели разумнее рассматривать как второй этап.

## 4. Оркестрация RAG pipeline

### Вариант 1. Свой простой 2-step RAG

Схема:

1. Подготовить и индексировать документы.
2. По запросу найти релевантные фрагменты.
3. Собрать prompt с контекстом.
4. Сгенерировать ответ.
5. Если релевантность низкая, предложить оставить заявку.

**Плюсы**

- максимальная прозрачность;
- проще дебажить и объяснять всей команде;
- меньше скрытой магии;
- лучше контролируется поведение бота.

**Минусы**

- часть glue-code придется писать вручную;
- ingestion, evaluation и observability нужно закладывать самостоятельно.

**Сложность освоения:** низкая-средняя.

### Вариант 2. LlamaIndex

**Плюсы**

- хорошо заточен под RAG;
- много готовых abstraction layers для индексации, retrieval и workflows;
- естественно ложится на Python-стек.

**Минусы**

- можно довольно быстро уйти в лишние абстракции для маленького проекта.

**Сложность освоения:** средняя.

### Вариант 3. LangChain + LangGraph

**Плюсы**

- огромная экосистема интеграций;
- удобен, если дальше появятся tools, CRM-коннекторы и agentic-сценарии.

**Минусы**

- высок риск переусложнения на старте;
- нужно дисциплинированно ограничивать архитектуру.

**Сложность освоения:** средняя-высокая.

### Вариант 4. Haystack

**Плюсы**

- сильная pipeline-модель;
- хорошие retriever/evaluation паттерны;
- подходит для более “платформенного” ML/backend подхода.

**Минусы**

- для малого бизнеса и MVP часто избыточен;
- выше порог входа по сравнению с простым собственным пайплайном.

**Сложность освоения:** средняя-высокая.

### Вывод по компоненту

На первом этапе рационально делать **свой 2-step RAG**.  
Если позже понадобится сложная retrieval-логика или много интеграций, можно перейти на **LlamaIndex** или **LangGraph**.

## Рекомендуемый стек

### Зафиксированное решение

- **Язык и бот-фреймворк:** `Python 3.12+` + `aiogram 3`
- **Способ работы без сервера на старте:** `long polling`
- **Основная БД:** `PostgreSQL`
- **Поиск по знаниям:** `pgvector + PostgreSQL FTS` с hybrid retrieval
- **LLM:** `OpenAI GPT-5 mini`
- **Embeddings:** `text-embedding-3-small`
- **RAG orchestration:** собственный `2-step RAG` на Python
- **Сбор заявок:** FSM/сцены в `aiogram`, хранение в Postgres
- **Уведомление владельца:** сообщение в личный Telegram-чат или админ-группу
- **Локальный запуск:** `Docker Compose` на локальной машине с автоперезапуском контейнеров

---

## Почему выбран именно этот стек

### 1. Понятен команде с разным уровнем опыта

Python остается самым практичным общим знаменателем для:

- Telegram-бота;
- обработки документов;
- embeddings/RAG;
- простого backend-кода;
- локального запуска.

Это уменьшает стоимость коммуникации и онбординга.

### 2. Минимум инфраструктурного зоопарка

PostgreSQL позволяет хранить:

- документы базы знаний;
- чанки документов;
- embedding-вектора;
- заявки клиентов;
- логи и техслужебные данные.

Это сильно упрощает MVP.

### 3. RAG остается контролируемым

Собственный 2-step pipeline проще:

- тестировать;
- ограничивать;
- дебажить;
- объяснять бизнесу.

Для customer support сценариев это часто важнее, чем “агентность”.

### 4. Есть хороший путь роста

Если база знаний и нагрузка вырастут, можно постепенно:

- вынести retrieval в Qdrant;
- добавить reranking;
- подключить более сложную оркестрацию;
- ввести отдельные worker-процессы;
- перевести запуск на сервер и webhook.

Базовое решение не блокирует развитие.

---

## Что не рекомендуется на первом этапе

- не начинать с **agentic RAG**;
- не тащить сразу тяжелый orchestration framework без явной необходимости;
- не делать локальную LLM единственным вариантом, если нет сильного privacy-требования;
- не смешивать сбор заявок и генерацию свободного ответа в один неконтролируемый prompt flow.

---

## Практический MVP-подход

### Рекомендуемая архитектура первого релиза

1. Пользователь пишет вопрос в Telegram.
2. Бот определяет: это вопрос по знаниям или пользователь хочет оставить заявку.
3. Для вопросов:
   - выполняется hybrid retrieval;
   - в LLM передается только найденный контекст;
   - если ответ неуверенный, бот предлагает оставить заявку.
4. Для заявок:
   - бот последовательно собирает `имя`, `телефон`, `вопрос`;
   - сохраняет заявку в БД;
   - отправляет уведомление владельцу.

### Минимальные сущности в БД

- `documents`
- `document_chunks`
- `leads`
- `admins`
- `message_logs`

## Альтернативы, если изменятся ограничения

### Если нужен самый быстрый локальный прототип

- `Python + aiogram + LanceDB + OpenAI`

### Если команда TypeScript-first

- `TypeScript + grammY + PostgreSQL/pgvector + OpenAI`

### Если нужен локальный privacy-first контур

- `Python + aiogram + Qdrant/LanceDB + Ollama + локальная модель`

Но это нужно считать отдельной веткой с более высоким риском по срокам и качеству.
